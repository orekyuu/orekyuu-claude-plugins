## テストの粒度・種別

テストピラミッドの考え方に従い、**単体テストを最も厚く**書く。

| 種別       | 対象                                | 量             |
| ---------- | ----------------------------------- | -------------- |
| 単体テスト | Domain 層・Application 層のロジック | 多（厚く書く） |
| 統合テスト | Infrastructure 層・レイヤー間の結合 | 中             |
| E2E テスト | ユーザーシナリオ全体                | 少             |

### 単体テストを厚くする理由

- 実行が速く、CI でのフィードバックループが短い
- 失敗箇所が明確で原因の特定が容易
- ビジネスロジックの正確さを細粒度で検証できる

### 単体テストで優先的にカバーすること

- ビジネスルールの正常系・異常系・境界値
- Result / Either 型の成功・失敗の両パス
- ドメインオブジェクトの状態遷移
- Parse 境界（不正な入力の拒否）

## テストの命名規則

テスト名は「XXXのとき、YYYになること」の形式で記述する。

「入力が空文字のとき、バリデーションエラーになること」

## テストケースの構造

テストケースは `// setup` `// act` `// assert` のコメントでブロックを分ける。

```ts
test("残高が不足しているとき、InsufficientBalanceError になること", () => {
  // setup
  const account = new BankAccount({ balance: 100 });

  // act
  const result = account.withdraw(200);

  // assert
  expect(result.ok).toBe(false);
  expect(result.error).toBeInstanceOf(InsufficientBalanceError);
});
```

- 各ブロックは1つの役割のみ持つ（setup で act しない、など）
- `// act` は原則1行にする。複数行になる場合は処理をメソッドに切り出すことを検討する
- `// assert` が複数になる場合も許容するが、検証対象が増えすぎる場合はテストを分割する

## 永続レイヤーの InMemory 実装

永続レイヤーのインターフェース（Repository など）は必ずインメモリ実装を用意する。

- Domain 層・Application 層の単体テストではインメモリ実装を使う。モックフレームワークは使わない
- インメモリ実装は `Map` などのオンメモリのデータ構造で状態を保持し、本番実装と同じインターフェースを満たす
- インメモリ実装自体もテストで検証される「テスト用の正しい実装」として扱う（モックとは異なる）

```
// 例: インターフェースとその実装
UserRepository (interface)
├── JpaUserRepository     ← 本番用（DB に永続化）
└── InMemoryUserRepository ← テスト用（Map で保持）
```

これにより、外部依存なしに高速な単体テストが書ける。また、テストダブルの設定ミスによる偽陽性を防げる。
