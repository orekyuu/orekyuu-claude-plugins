# 設計原則

このドキュメントは `sdd-impl` エージェントが実装時に参照する設計原則を定義します。
プロジェクトの方針に合わせて内容を編集してください。

## アーキテクチャ原則

このプロジェクトのアーキテクチャは以下の原則を柱とする。

- レイヤーを明確に分割し、依存は外側から内側（Domain 層）へ一方向に保つ
- パッケージはレイヤーではなくドメインの単位で切る
- IO・永続レイヤーはインターフェースで抽象化し、Domain 層をフレームワークから独立させる

各原則の詳細は以降のセクションを参照すること。

## レイヤー構成と依存の方向

アーキテクチャを以下のレイヤーに分割し、**依存は必ず外側から内側（ドメイン層）へ向かう**方向に保つ。

```
┌─────────────────────────────────┐
│  Presentation / Adapter 層      │  HTTP・CLI・イベントハンドラなど
├─────────────────────────────────┤
│  Application 層                 │  ユースケース・コマンド・クエリ
├─────────────────────────────────┤
│  Domain 層（フレームワーク非依存）│  ビジネスロジック・ドメインモデル
├─────────────────────────────────┤
│  Infrastructure 層              │  DB・外部API・メッセージング
└─────────────────────────────────┘
         依存の方向: 外 → 内
```

### Domain 層の定義

Domain 層はアプリケーションの中核であり、**いかなるフレームワーク・ライブラリにも依存しない**ピュアなコードで構成する。

- フレームワークのアノテーション（`@Entity`、`@Component` など）を含めない
- ORMやHTTPクライアントなどの具体実装をインポートしない
- 標準ライブラリのみに依存する形を理想とする

### 依存性逆転の適用

Domain 層が Infrastructure 層の機能（DB・外部API など）を必要とする場合は、依存性逆転の原則（DIP）を適用する。

1. Domain 層にインターフェース（ポート）を定義する
2. Infrastructure 層がそのインターフェースを実装する（アダプター）
3. DI によって実行時に具体実装を注入する

```
Domain 層                        Infrastructure 層
┌──────────────────┐             ┌────────────────────────┐
│ UserRepository   │ ←実装─────  │ JpaUserRepository      │
│ (interface)      │             │ (具体実装)              │
└──────────────────┘             └────────────────────────┘
依存の方向: Infrastructure → Domain（逆転している）
```

### ルール

- Domain 層のクラスが外側のレイヤーをインポートしていたらアーキテクチャ違反と判断する
- Application 層はドメインオブジェクトを組み合わせてユースケースを表現し、ドメインロジック自体は持たない
- フレームワーク固有の処理（トランザクション管理・バリデーションアノテーションなど）は Adapter / Infrastructure 層に留める

## パッケージ構成

パッケージはレイヤー（`entity` / `domain` / `repository` など）ではなく、**ドメインの単位**で切る。

```
# 悪い例（レイヤー単位）
src/
├── entity/
│   ├── User.java
│   └── Order.java
├── repository/
│   ├── UserRepository.java
│   └── OrderRepository.java
└── service/
    ├── UserService.java
    └── OrderService.java

# 良い例（ドメイン単位）
src/
├── users/
│   ├── User.java
│   ├── UserRepository.java
│   └── UserService.java
└── checkout/
    ├── Order.java
    ├── OrderRepository.java
    └── CheckoutService.java
```

### 理由

- 関連するコードが同じパッケージにまとまり、凝集度が高くなる
- あるドメインを変更する際に影響範囲が1パッケージ内に収まりやすい
- パッケージ名がドメイン用語に対応するため、コードの意図が読み取りやすい

### ルール

- パッケージ名はドメインの概念を表す名詞にする（`users`、`checkout`、`inventory` など）
- 複数ドメインにまたがる共通処理は `shared` または `common` パッケージに置く
- ドメイン間の依存は最小化し、依存が必要な場合はインターフェースを介する

## コーディング規約

- **メソッドの粒度**: 1メソッド1責務。処理が複数の「段階」に分かれる場合はメソッドを分割する
- **クラスの粒度**: 1クラス1責務（SRP）。名前が思い浮かばない・テストが書きにくいときは分割のサイン
- **コメントの方針**: 「何をしているか」ではなく「なぜそうしているか」を書く。コードで表現できる内容はコメントにしない
- **マジックナンバー・マジック文字列の禁止**: 定数・Enum・ドメイン型に切り出す

## 命名規則: 曖昧なサフィックスを避ける

`Util`・`Manager`・`Helper`・`Service` などの曖昧なサフィックスを避け、クラスの責務が明確に伝わる名前をつける。

### 避けるべきサフィックスと代替例

| 避けるべき名前  | 問題点                           | 代替候補                                                      |
| --------------- | -------------------------------- | ------------------------------------------------------------- |
| `UserManager`   | 「管理する」が何をするか不明     | `UserRegistry`（登録・検索）、`UserSession`（セッション管理） |
| `OrderService`  | 何でも入る万能クラスになりやすい | `OrderPlacement`（注文受付）、`OrderFulfillment`（履行処理）  |
| `StringUtil`    | 雑多な処理の置き場になりやすい   | 処理の意味に応じたドメイン型やメソッドに移動する              |
| `PaymentHelper` | 補助的という意味しか伝わらない   | `PaymentGateway`（決済連携）、`PaymentPolicy`（決済ルール）   |

### 使いやすい代替サフィックス

- `Registry` — オブジェクトの登録・検索・管理
- `Catalog` — 一覧・定義の集合
- `Policy` — ビジネスルール・条件の判定
- `Factory` — オブジェクトの生成ロジック
- `Gateway` — 外部システムとの接続口
- `Handler` — 特定イベント・リクエストの処理
- `Resolver` — 値・依存の解決ロジック
- `Validator` — 入力の検証（ただし Parse, don't validate を先に検討する）

### ルール

- クラス名だけを見て「何をするクラスか」が分かることを基準にする
- 名前が思い浮かばないときは責務が広すぎるサインと捉え、クラスの分割を検討する

## 設計パターン

以下のパターンを積極的に採用する。コードレビューではこれらのパターンに沿っているかを確認すること。

- **Tell, Don't Ask** — ロジックをデータのそばに置く（詳細は以降のセクション）
- **Parse, don't validate** — 不正な状態を型で表現不可能にする（詳細は以降のセクション）

新たなパターンを導入する場合はチームで合意し、このドキュメントに追記すること。

## Tell, Don't Ask

オブジェクトの状態を外部から取得して判断するのではなく、**オブジェクト自身に振る舞いを命令する**。
ロジックはデータを持つオブジェクトの中に置く。

### 悪い例（Ask）

```java
// 外部からオブジェクトの状態を聞いて、呼び出し側が判断している
if (order.getStatus() == Status.PENDING && order.getAmount() > 0) {
    order.setStatus(Status.CONFIRMED);
    payment.charge(order.getAmount());
}
```

### 良い例（Tell）

```java
// オブジェクトに命令するだけ。判断と状態変更はオブジェクト自身が行う
order.confirm(payment);
```

### ルール

- ゲッターで状態を取り出して外側で条件分岐するコードは Tell, Don't Ask 違反を疑う
- ドメインオブジェクトのビジネスルールはそのオブジェクト自身のメソッドとして定義する
- 「〜かどうか確認してから〜する」という処理は「〜する（オブジェクト内部で確認する）」に書き直せないか検討する
- ただしクエリ（読み取り専用の参照）は許容する。状態を取得して**外部でロジックを書く**ことを避けるのが目的

## Parse, don't validate

> 参考: [Parse, don't validate](https://lexi-lambda.github.io/blog/2019/11/05/parse-don-t-validate/)

入力値の検証は「条件を確認して true/false を返す」のではなく、**より厳密な型への変換（パース）として表現する**。
不正な状態を型システムで表現不可能にすることで、バリデーションの抜け漏れをコンパイル時に排除する。

### 考え方

- **検証（validate）**: 条件を満たすか確認するが、その情報は型に残らない。呼び出し側で何度もチェックが必要になる
- **解析（parse）**: 生の入力を受け取り、制約を型として表現した値に変換する。変換後は制約が保証される

### ルール

- システム境界（HTTPリクエスト・ファイル入力・外部APIレスポンスなど）でのみパース処理を行い、内部では信頼された型のみを扱う
- プリミティブ型（`String`、`int` など）をそのままドメインロジックに渡さず、意味を持つ型（`Email`、`UserId`、`NonEmptyList` など）に変換する
- スマートコンストラクタ（ファクトリメソッドや `of()` など）でインスタンス化し、不正な値を持つオブジェクトが生成できないようにする
- ありえないケースを `null` チェックや例外で握りつぶすのではなく、型設計を見直して「そもそも表現できない」状態にする

### 例

```
// 悪い例: String のまま渡し、各所でバリデーション
void register(String email) {
    if (!email.contains("@")) throw new IllegalArgumentException();
    // ...以降も email が String のまま流れる
}

// 良い例: 境界でパースし、内部では Email 型のみ使う
Email email = Email.parse(rawInput); // 不正な場合はここで失敗
void register(Email email) {
    // Email が渡ってきた時点で正当性が保証されている
}
```

## IO・永続レイヤーの抽象化

IO（ファイル・ネットワーク・外部APIなど）および永続レイヤー（DB・キャッシュなど）は必ずインターフェースで抽象化し、ビジネスロジックが具体実装に直接依存しない設計にする。

### ルール

- リポジトリ・クライアント・ストレージなどの IO 境界はインターフェースとして定義する
- ビジネスロジック（サービス・ユースケース）はインターフェースのみに依存し、具体実装をインポートしない
- 具体実装はDI（依存性注入）で差し替えられるようにする

### 理由

- テスト時にインメモリ実装やモックに差し替えられるため、外部依存なしに単体テストが書ける
- 将来的に永続化手段（RDB → NoSQL など）を変更してもビジネスロジックへの影響がない

### 例

```
# ドメインパッケージ内の構成例
users/
├── UserRepository.java        ← インターフェース（ビジネスロジックが依存する）
├── UserService.java           ← UserRepository のみに依存
├── InMemoryUserRepository.java ← テスト用実装
└── JpaUserRepository.java     ← 本番用実装（infrastructure 層に置く場合もある）
```

## エラーハンドリング方針

**例外の伝播ルール:**

- Domain / Application 層では例外をキャッチして握りつぶさない。呼び出し元に伝播させる
- Presentation / Adapter 層でのみ例外をキャッチし、エラーレスポンス・ログ出力に変換する
- チェック例外（Java の checked exception）はインターフェース境界を越えないよう非検査例外に変換する

**ログの粒度:**

- ERROR: 予期しない例外・システム障害（即時対応が必要なもの）
- WARN: 回復可能なビジネスエラー・リトライが発生した場合
- INFO: ユースケースの開始・完了など操作ログ
- DEBUG: 開発時のデバッグ情報。本番では出力しない

### エラーの種別と対応方針

エラーを以下の3種別に分類し、それぞれ異なる手段で表現する。

| 種別                     | 対応                           | 例                                |
| ------------------------ | ------------------------------ | --------------------------------- |
| 引数の不正               | 例外（プログラムのバグ）       | `null` 渡し、型違反、前提条件違反 |
| 外部システムエラー       | 例外（呼び出し側では対処不能） | DB接続失敗、外部APIタイムアウト   |
| 回復可能なビジネスエラー | Either / Result 型             | 残高不足、在庫切れ、重複登録      |

### ドメインロジックでの Either / Result 型

ビジネスルールの結果として「成功」または「失敗（回復可能）」があり得る場合は、戻り値を Either / Result 型で表現する。

- 呼び出し側はコンパイル時にエラーケースの処理を強制される
- 例外のように暗黙的に伝播せず、エラーの発生箇所と処理箇所が明示される
- ドメイン固有のエラー型（`InsufficientBalanceError`、`OutOfStockError` など）を定義してエラーの意味を型に持たせる

```java
// 例: 成功 or 業務エラーを Result 型で返す
Result<Order, OrderError> checkout(Cart cart, Payment payment);

// 呼び出し側はエラーを無視できない
checkout(cart, payment)
    .onSuccess(order -> ...)
    .onFailure(error -> ...);
```

### 例外を使うケース

- **引数の不正**: `null`・不正な値が渡された場合はプログラムのバグであるため例外を投げる（`IllegalArgumentException` など）。Parse, don't validate と組み合わせ、ドメイン型のスマートコンストラクタで入口を塞ぐことを優先する
- **外部システムエラー**: DB・外部API・ファイルシステムなどのエラーは Infrastructure 層で例外として扱い、呼び出し側での個別対処を想定しない
